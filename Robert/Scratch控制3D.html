Scratch

Scratch是一款由麻省理工学院(MIT) 设计开发的一款面向少年的简易编程工具。 这个软件的特点是：使用者可以不认识英文单词，也可以不会使用键盘。构成程序的命令和参数通过积木形状的模块来实现。用鼠标拖动模块到程序编辑栏就可以了。 ——百度百科

Scratch的操作界面如图所示

通过简单的积木式拼接，就可以完成编程。没碰过编程的人即使在不看教程的情况下也可以用Scratch做点小程序。

然而，用久了Scratch的小伙伴会发现，Scratch实现的功能实在太少了，所做的成品也只能是2D的。虽然国外的Scratch大神已经可以通过Scratc本身的画图工具实现了3D，但效率太低，效果也比较差。

Scratch中实现的3D效果

我刚玩Scratch不久，就盼望着Scratch可以实现更多功能。首先，我用了歪门邪道的方法开发了Scratch3D，然后开发出了Scratch+。

Scratch+

Scratch+的原理是通过解析Scratch的工程文件，读取里面的json格式的内容，然后把Scratch的代码块翻译成C#代码，再编译运行之。虽然运行效率有了，功能有了，但可使用性范围小，不能很好地融入到其他已存在的Scratch项目中；开发难度大，能够使用Scratch+的人想必也对C#了解不少。

不仅如此，Scratch开发者们也意识到了Scratch的稀缺功能，于是他们开发了模块扩展功能，可以利用Scratch控制Scratch以外的程序，主要是通过HTTP通信实现的。

Codeic

于是，7月1日，我发布了Codeic——一个可扩展Scratch功能的软件。Codeic正是利用了Scratch的扩展模块功能。

Codeic是一款基于Scratch或其他程序的拓展程序，为Scratch提供了多种接口，并且Codeic可支持扩展，使Scratch可以通过Codeic调用更多接口、程序，使Scratch的应用范围更加广泛。

Codeic可以实现原版Scratch无法实现的功能，比如对系统磁盘进行操作、让电脑根据文本发出语音、获取网页的内容，同时Codeic还提供了许多工具包，使Scratch的功能更加完善。

简单来说，Scratch可以通过Codeic这一平台与更多的组件或程序进行交互。

这里，我们要来实现控制3D的效果，为了方便，我打算用Unity开发实现3D效果的扩展程序，然后让它可以通过Codeic与Scratch交互。

配置开发环境

首先，我们需要安装Codeic，点击这里查看Codeic安装教程。如果想要了解Codeic，点击这里查看Codeic官方教程，里面介绍了详细的Codeic使用方法和Codeic原理解析。

然后，我们需要安装Scratch离线版，点击这里查看下载。

Codeic可以调用扩展库或扩展程序（想要了解它们的区别请看这里）。本例我们将开发一个扩展程序供Codeic调用。

扩展程序需要依靠一个叫做CodeicRuntime的dll文件，事实上，Codeic本身也需要依靠CodeicRuntime，CodeicRuntime是Scratch与Codeic交互的核心，里面处理着各种交互功能，所以，我们的程序需要引用这个dll。除此之外，还需要添加CDK的dll，它是CodeicRuntime和程序本身的桥梁，没有它，你的CodeicRuntime也没法和相关的3D接口交互。

CodeicRuntime和CDK哪里来？可以直接在Codeic的主目录下获取。

接下来创建一个Unity项目。然后我们就要把CodeicRuntime和CDK这两个dll导入进Unity中。首先在Assets文件夹下创建一个叫做Plugins的文件夹，然后把这两个dll都拖进去，完成。

为了让Unity里的场景不那么单调，可以导入一些资源包，随便搞搞。

然后，打开VS，开始搞事情吧！

让Unity与Scratch相遇

我们想要的效果是：Scratch告诉Codeic去启动我们开发的扩展程序（名之为Codeic3D吧），Codeic启动Codeic3D后告诉我们的程序：我已经帮你搭好桥了，接下来你自己去和Scratch爽爽吧，我走了，拜拜！Codeic退出了。然而，Scratch早已知道Codeic3D会在哪里等他，我们这里设置为8001端口（可以随意设置，不冲突即可），于是，Scratch就会在8001端口呼喊：Codeic3D！你在哪里？你在哪里？我在等你呢？你听得到吗？Scratch呼喊的频率最快可以达到每秒30次。Codeic唤醒沉睡的Codeic3D后，Codeic3D就去8001端口找Scratch了，然后开始交互。

前戏我们准备好了，接着就开始玩玩CodeicRuntime和CDK。

先来看看CodeicRuntime都有些什么。

CodeicRuntime的类

然而我们只需要了解Runner这个类。Runner顾明思义，就是一个运行器，它是CodeicRuntime的核心，大部分工作靠它完成。另外还有一个乱入的Essential类，虽然名字听起来很霸气，然而我们不需要用它。

你也许会有疑问了，为什么不需要用它，它却还在CodeicRuntime里？没错，这不是我的疏忽大意。我们可以悄悄看看Essential里到底是个啥。


using CDK;

namespace Codeic
{
    public class Essential : ILib
    {
        public IRunner runner;
    
        public Essential(IRunner runner);
    
        public string author { get; }
        public string name { get; }
        public string version { get; }
    
        public void Awake(IRunner runner);
        public void Receive(string command, string[] parameters);
        public void Reset();
        public void Start();
    }
}

没想到，它竟然继承了ILib接口！ILib是CDK的东西，继承ILib接口的一般是一个扩展库，也就是说，Essential是一个扩展库！它就是基础组件，内置在了CodeicRuntime中。它的功能是自动安装某个库或扩展程序，也就是说，通过它，我们自己写的扩展程序也可以像Codeic那样再次读取自己的扩展库或扩展程序。不过，一般是比较复杂庞大的扩展程序才需要用它。我们现在就可以忽略它。

设置Runner


using UnityEngine;
using Codeic;

public class C3D : MonoBehaviour {

    // Use this for initialization
    void Start () {
        Runner runner = new Runner("localhost",8001);
        runner.Start();
    }
    
    // Update is called once per frame
    void Update () {
        
    }
}

我们创建了一个脚本并绑定在了一个物体上，脚本文件如上。

Runner runner = new Runner("localhost",8001);
runner.Start();

在这里，我们设置了侦听的IP和端口，也就是Scratch与Codeic3D相约的地方。运行Start方法时Codeic3D就开始侦听了。

然而，需要注意的是，运行Start方法可是会阻塞的！如果直接运行这个脚本，那么这个程序就会被阻塞，在Unity里运行会导致Unity卡住。

所以我们需要新建一个线程，来运行这个Runner。所以，代码修改如下：

using UnityEngine;
using Codeic;
using System.Threading;

public class C3D : MonoBehaviour {

    // Use this for initialization
    void Start () {
        Runner runner = new Runner("localhost", 8001);
        Thread t = new Thread(new ThreadStart(runner.Start));
        t.Start();
    }

    // Update is called once per frame
    void Update () {
        
    }
}

可是，Scratch怎么知道是在这个地方相约呢？这就需要扩展文件了。

扩展文件是json格式的文本文件，描述了相约的地点，相约后的动作，相约的规则。如下

{
    "extensionName": "Codeic3D",//扩展文件的名字，可随意改变
    "extensionPort": 8001,//Codeic3D侦听的端口
    "host": "localhost",
    "useHTTP": true,
    "blockSpecs":[

    ]
}

现在我们来测试一番，运行Unity，给Scratch添加这个扩展文件，点击查看添加扩展文件教程。

可以看到，在扩展模块的Codeic3D扩展竟然亮绿灯啦，这说明他们交互了！

但，你有没有发现有什么不对？我们刚刚测试的时候，没有用到Codeic！事实上，带着CodeicRuntime就可以有Codeic类似的功能了，Codeic所起的不过是拉线搭桥的作用。相反，如果没有靠Codeic来拉线搭桥，那完成这个交互需要双方，也就是Codeic3D和Scratch都运行的情况下，有了Codeic，Codeic就会帮忙启动Codeic3D。

创建扩展库

可以看到，我们的扩展还没有任何功能。所以我们需要添加扩展库。没错，扩展程序同样也有扩展库。

        {

        {

        {
            
        }

这时你会注意到，扩展库的名字竟然用的是Essential！没错，我们篡位了！我们不需要以前的Essential，但Essential有个天生的优势啊：调用它的命令可以不需要任何前缀！待会我来解释为什么。如果你高兴，你也可以自己起个名字。所以，实际上，这是一个我们自己写的Essential库。

然后我们把它加进Runner，在Runner运行前加入如下代码：

runner.AddLib(new Codeic3D());

就这样，我们把该实现的方法都实现了，现在还是添加功能。

添加功能

实现了Unity和Scratch的交互，接着就来添加功能。我们可以添加各种各样的功能，这里来做个简单的示范，我们就添加一个在某个位置生成一个物体的功能，其他功能都可以通过这个示例实现出来。

编写扩展文件

扩展文件，我们又见面了，上一次见到你，你还没有任何代码块，所以来添加一个代码块吧。

{
    "extensionName": "Codeic3D",
    "extensionPort": 8001,
    "host": "localhost",
    "useHTTP": true,
    "blockSpecs":[
        [" ","在X= %n Y= %n Z= %n 的地方生成一个物体","CreateObject"]
    ]
}

哈哈，blockSpecs里有东西里，是个代码块。重新导入到Scratch：

格式显而易见，有三个参数，第一个“ ”表明这是一个普通代码块，第二个是显示样式，%n说明这里是个整数，%s说明这里是个字符串，然而我们并没用到，第三个参数就是命令。注意了！这个命令看起来很简洁，但前提是刚刚创建的扩展库的名字要为Essentials，假如你刚刚一高兴，自己起了个名字比如MyOwnLib，那么这个命令你就要写成MyOwnLib:CreateObject，要添加个前缀哟！

在Codeic3D里实现功能

把刚刚创建的扩展库的Receive方法写成这样：

        public void Receive(string command, string[] parameters)
        {
            switch (command)
            {
                case "CreateObject": Debug.Log(string.Format("在X={0}，Y={1}，Z={2}的位置创建了个物体", parameters[0], parameters[1], parameters[2]));
                    break;
            }
        }

现在不过是加了个Debug.Log方法，代码块的参数依次存放在parameters字符串数组里，一个一个取出来。测试一番吧。

成功啦！然而，不要忽略一个问题哟，我们这里可是在新开的线程里和Scratch交互的哦，而Unity主线程对此一无所知！所以，我们要把Scratch传过来的命令告诉Unity主线程。
